Início rápido

O fluxo do dia a dia em linguagem direta. Nada de instalar ou configurar — só o que você usa toda vez que vai trabalhar.

git init: só uma vez por projeto. Ele cria o repositório Git na pasta. Se você já rodou antes, não precisa rodar de novo — nunca. Para confirmar que a pasta já é um repo, rode git status. Se aparecer informações sobre branch e arquivos, já está pronto.

Antes de começar a trabalhar
Sequência de abertura do projeto
Sempre que for trabalhar no projeto — seja hoje, amanhã ou daqui a uma semana — o fluxo de entrada é este:
%cd nome-do-projeto
Entra na pasta do projeto
%git pull
Baixa tudo que há de novo no GitHub antes de começar
%code .
Abre o VS Code na pasta atual

Sem git init. Sem git status antes do pull. Só cd → git pull → code . — e você já pode trabalhar.

Durante o trabalho
O ciclo que se repete o dia inteiro
Faça isso quantas vezes quiser no dia. Cada commit é um ponto de salvamento no histórico.
%git status
Veja o que mudou desde o último commit
%git add .
Prepara todas as mudanças para o commit
%git commit -m "feat: descrição clara do que fez"
Salva no histórico local com uma mensagem descritiva
%git push
Envia para o GitHub
Resumo visual
O fluxo completo em uma linha
Momento Comando Por quê
Abrir o projeto git pull Sincroniza com o GitHub antes de qualquer coisa
Verificar estado git status Vê o que mudou e o que está no stage
Preparar mudanças git add . Move as mudanças para a staging area
Salvar no histórico git commit -m "msg" Cria o ponto de salvamento local
Publicar git push Envia os commits para o GitHub

Regra de ouro: git pull ao abrir. git push ao fechar. Commits pequenos e frequentes no meio.

Comandos similares — qual usar?

Git tem vários comandos que parecem fazer a mesma coisa. Aqui estão as diferenças que realmente importam no dia a dia.

Adicionar ao stage
git add . vs git add -A vs git add arquivo
Comando O que faz Quando usar
git add . Adiciona tudo a partir da pasta atual Uso geral — funciona perfeitamente se você está na raiz do projeto
git add -A Adiciona tudo no repositório inteiro, independente de onde você está no terminal Projetos com estrutura de pastas complexa onde você pode estar numa subpasta
git add arquivo.txt Adiciona só aquele arquivo específico Quando quer commitar mudanças de um arquivo sem levar as outras
git add -p Modo interativo — você aprova trecho por trecho de cada arquivo Um arquivo tem duas mudanças não relacionadas e você quer commitar só uma

Na prática: se você sempre roda os comandos da raiz do projeto — o que é o normal — git add . e git add -A são equivalentes. Use git add . no dia a dia.

Baixar mudanças
git pull vs git fetch
Comando O que faz Quando usar
git pull Baixa as mudanças do GitHub E aplica no seu código imediatamente (fetch + merge em um comando) No início de toda sessão de trabalho — é o uso mais comum
git fetch Baixa as mudanças mas não aplica — fica guardado esperando você decidir Quando quer ver o que mudou no GitHub antes de decidir se aplica. Útil em times grandes antes de um merge delicado.

Após um git fetch, você pode rodar git log origin/main para ver os commits que vieram, e depois git merge origin/main para aplicar. Mas no dia a dia, git pull faz tudo isso de uma vez.

Mesclar branches
git merge vs git rebase
Comando O que faz Resultado no histórico
git merge Integra as mudanças criando um commit de merge. Preserva o histórico exato — quando cada coisa aconteceu. Histórico fiel à realidade, com bifurcações visíveis
git rebase Reescreve seus commits como se tivessem sido feitos em cima da outra branch. Histórico mais linear e limpo. Histórico linear, sem bifurcações — parece que tudo foi feito em sequência

Regra de ouro do rebase: nunca use git rebase em branches compartilhadas com outras pessoas — ele reescreve o histórico e causa conflitos sérios para quem já tem os commits. Use rebase só em branches locais suas. Para quem está começando: use git merge sempre.

Desfazer commits
git reset --soft vs --mixed vs --hard
Comando Desfaz o commit? O que acontece com as mudanças?
git reset --soft HEAD~1 Sim As mudanças ficam no stage — prontas para um novo commit
git reset --mixed HEAD~1 Sim As mudanças ficam nos arquivos mas saem do stage — você precisa fazer add de novo
git reset --hard HEAD~1 Sim As mudanças são descartadas — os arquivos voltam ao estado do commit anterior. Irreversível.

--hard é destrutivo. Código não commitado desaparece para sempre. Use --soft quando quiser desfazer o commit mas guardar o trabalho.

Trocar de branch
git switch vs git checkout
Comando O que faz Status
git switch nome Troca de branch Moderno — prefira este
git switch -c nome Cria e troca de branch Moderno — prefira este
git checkout nome Troca de branch (e outras coisas) Legado — ainda funciona
git checkout -b nome Cria e troca de branch Legado — ainda funciona

git checkout era o comando antigo que fazia tudo: trocava branch, restaurava arquivos, criava branches... Justamente por fazer demais, o Git criou git switch (só branches) e git restore (só arquivos). São mais claros e explícitos. Você vai ver muito checkout em tutoriais antigos — funciona, mas o switch é o caminho atual.

Reverter mudanças
git revert vs git reset
Comando O que faz Quando usar
git revert HEAD Cria um novo commit que desfaz o anterior. O histórico original é preservado. Branches compartilhadas, código já publicado no GitHub — é a forma segura
git reset HEAD~1 Apaga o commit do histórico, como se nunca tivesse existido. Branches locais, commits que ainda não foram para o GitHub

Regra simples: se o commit já foi para o GitHub, use git revert. Se ainda está só na sua máquina, pode usar git reset.

Fluxos para criar um repositório

Escolha o fluxo de acordo com como você vai começar o projeto.

Fluxo A — GitHub primeiro Recomendado
Cria o repositório no GitHub e clona para o Mac. O mais simples, sem conflitos.
Fluxo B — Terminal primeiro
Inicia o Git na pasta local e conecta ao GitHub depois. Útil quando o projeto já existe.
Fluxo C — 100% terminal
Cria o repositório no GitHub via CLI sem abrir o navegador.

Quando usar: sempre que for começar um projeto do zero. É o fluxo mais limpo pois evita conflitos entre arquivos locais e remotos.

Atenção: ao criar o repositório no GitHub, não marque nenhuma opção de README, .gitignore ou licença — deixe o repositório totalmente vazio.

1
Crie o repositório no GitHub (vazio)
Acesse github.com → botão New → dê um nome → clique em Create repository sem marcar nenhuma opção.
2
Clone o repositório para o Mac
%git clone git@github.com:usuario/nome-do-repo.git
Isso cria a pasta do projeto já conectada ao GitHub
3
Entre na pasta e adicione seus arquivos
%cd nome-do-repo
Entre na pasta clonada e copie seus arquivos para cá
4
Publique no GitHub
%git add .
%git commit -m "primeiro commit"
%git push
Como clonou, o push já sabe o destino — sem precisar de -u origin main

Projeto publicado! A partir de agora o ciclo é sempre: git add . → git commit -m "msg" → git push

Quando usar: quando o projeto já existe na sua máquina e você quer versioná-lo no GitHub agora.

Atenção: ao criar o repositório no GitHub, não marque nenhuma opção de README, .gitignore ou licença — repositório completamente vazio.

1
Entre na pasta do projeto e inicie o Git
%cd ~/caminho/para/seu-projeto
%git init
Cria o repositório Git local na pasta
2
Crie o repositório no GitHub (vazio)
Acesse github.com → botão New → dê um nome → clique em Create repository sem marcar nenhuma opção.
3
Conecte o repositório local ao GitHub
%git remote add origin git@github.com:usuario/nome-do-repo.git
4
Adicione os arquivos, faça o commit e publique
%git add .
%git commit -m "primeiro commit"
%git push -u origin main
O -u só é necessário no primeiro push — depois basta git push

Projeto publicado! Nas próximas vezes: git add . → git commit -m "msg" → git push

Quando usar: quando você quer fazer tudo pelo terminal sem abrir o navegador. Requer o GitHub CLI (gh) instalado.

1
Instale o GitHub CLI (se ainda não tiver)
%brew install gh && gh auth login
Instala e autentica com o GitHub
2
Entre na pasta do projeto e inicie o Git
%cd ~/caminho/para/seu-projeto
%git init
3
Faça o primeiro commit
%git add .
%git commit -m "primeiro commit"
4
Crie o repositório no GitHub e publique — tudo em um comando
%gh repo create nome-do-repo --public --source=. --remote=origin --push
Cria o repo no GitHub, conecta e faz o push em um só comando
Troque --public por --private para repositório privado.

Projeto criado e publicado sem abrir o navegador! Nas próximas vezes: git add . → git commit -m "msg" → git push

Fundamentos do Git

O ciclo básico de trabalho: adicionar arquivos, commitar e repetir.

Conceito: Git rastreia alterações em 3 áreas — working directory (seus arquivos), staging area (o que vai no próximo commit) e repository (histórico confirmado).

Do Treinamento
Fluxo do zero ao repositório — estilo Aulão da Javanauta
O treinamento usa um fluxo simplificado e direto ao ponto, ótimo para quem está começando. Os comandos abaixo seguem exatamente o que foi ensinado no material.
%mkdir meu-projeto
Cria a pasta do projeto
%cd meu-projeto
Entra na pasta
%git init
Transforma a pasta em repositório Git
%git status
Veja o estado dos arquivos no projeto
%git add .
Adiciona todos os arquivos ao stage
%git add nome-do-arquivo
Adiciona um arquivo específico ao stage
%git commit -m "sua mensagem"
Registra as mudanças com uma descrição clara do que foi feito
%git push -u origin main
Envia para o GitHub — o -u é necessário somente no primeiro push
%git pull origin main
Baixa e mescla as últimas mudanças do repositório remoto

Por que o treinamento usa git push -u origin main e não só git push?
O treinamento ensina o fluxo completo explícito, deixando claro o destino (origin) e a branch (main). O -u configura o upstream no primeiro push — depois disso, git push sozinho já funciona. Da mesma forma, o treinamento usa git pull origin main explícito para deixar óbvio de onde está puxando. Este guia usa a versão curta (git push) depois que o upstream já foi configurado — ambos são corretos.

Status e diff
Verificar o estado do repositório
%git status
Mostra arquivos modificados e staged
%git diff
Mostra diferenças em arquivos modificados
%git diff --staged
Mostra diferenças em arquivos staged

Adicionar arquivos ao stage
%git add arquivo.txt
Adiciona um arquivo específico

%git add -p

%git rm --cached arquivo.txt

Salvar alterações no histórico

Salva com mensagem em uma linha
%git commit -am "mensagem"

%git commit --amend -m "nova mensagem"
Edita o último commit
Ignorar arquivos
Criar .gitignore

Ignora arquivos do macOS

Use gitignore.io para gerar .gitignore adequado para sua stack.

Branches permitem trabalhar em funcionalidades separadas sem afetar o código principal.

Fluxo recomendado: mantenha main sempre estável. Crie uma branch por funcionalidade, faça commits lá e depois mescle com main.

Desfazer alterações

Corrija erros em diferentes estágios do fluxo.

Regra de ouro: nunca reescreva histórico de commits já enviados ao GitHub e usados por outros. Para isso, use revert.

Working directory
Desfazer mudanças não commitadas
%git restore arquivo.txt
Descarta mudanças locais do arquivo
%git restore --staged arquivo.txt
Remove arquivo do stage
Commits locais
Desfazer commits não publicados
%git reset --soft HEAD~1
Desfaz commit, mantém mudanças staged
%git reset --hard HEAD~1
Desfaz commit e descarta mudanças
Commits publicados
Reverter sem reescrever histórico Seguro
%git revert HEAD
Cria novo commit que desfaz o anterior
Emergência
Recuperar commits "perdidos"
%git reflog
Mostra histórico de movimentos do HEAD
%git switch -c recuperado abc1234
Recupera commit específico em nova branch

O reflog guarda tudo por 90 dias. Se fez um reset --hard errado, o commit ainda está lá!

Estratégias de Branch & Git Flow

Como organizar branches em equipes reais — desde projetos simples até fluxos profissionais com releases e hotfixes.

Conceito: Uma estratégia de branch define regras de como a equipe cria, nomeia e mescla branches. Sem regras, o repositório vira caos. Com regras, qualquer dev sabe exatamente onde trabalhar.

Branches principais — do treinamento
As 5 branches do fluxo profissional
O treinamento Javanauta apresenta exatamente este conjunto de branches como base de um fluxo profissional:
Branch Papel Quando usar
main / master Código estável e em produção Nunca commite direto — só recebe merges
develop Integração das funcionalidades Base de trabalho do time; sempre integra features aqui primeiro
feature/* Desenvolvimento de novas funcionalidades Uma branch por feature — ex: feature/login-social
release/* Preparação para novas versões Estabiliza código antes de ir para main — ex: release/1.2.0
hotfix/* Correções urgentes em produção Parte da main, vai direto para main e develop — ex: hotfix/crash-login
Ciclo feature
Fluxo completo de uma nova funcionalidade
Este é o ciclo que o treinamento chama de "Repetir o ciclo": branch → alterações → commit → push → pull request → merge.
%git switch develop
Parta sempre de develop para criar features
%git switch -c feature/nome-da-feature
Cria e entra na branch da feature
%git add . && git commit -m "feat: descrição da mudança"
Trabalhe e commite normalmente
%git push -u origin feature/nome-da-feature
Sobe a branch para o GitHub — abre Pull Request depois

Depois do push, abra um Pull Request no GitHub para mesclar sua feature em develop. Nunca em main diretamente!

Hotfix urgente
Corrigir um bug em produção sem esperar o ciclo normal
%git switch main
Hotfix parte de main (não de develop)
%git switch -c hotfix/descricao-do-bug
Cria branch de correção urgente
%git add . && git commit -m "fix: corrige crash no login"
Faz a correção e commita
%git switch main && git merge hotfix/descricao-do-bug
Mescla o hotfix em main
%git switch develop && git merge hotfix/descricao-do-bug
Mescla o hotfix em develop também para não perder a correção

Importante: sempre mescle o hotfix tanto em main quanto em develop — caso contrário a correção será perdida no próximo ciclo de release.

Convenção de nomes
Commits semânticos — padrão do mercado
Mensagens de commit padronizadas (como o treinamento recomenda) facilitam o histórico, changelogs automáticos e integração com CI/CD.
Prefixo Quando usar
feat: ... Nova funcionalidade — ex: feat: adiciona login social
fix: ... Correção de bug — ex: fix: corrige cálculo de frete
docs: ... Mudanças em documentação — ex: docs: atualiza README
style: ... Formatação, sem mudança de lógica — ex: style: indenta componente
refactor: ... Refatoração sem fix nem feature — ex: refactor: extrai serviço de auth
test: ... Adição ou correção de testes — ex: test: testa fluxo de pagamento
chore: ... Tarefas de manutenção — ex: chore: atualiza dependências

CI/CD — Integração e Entrega Contínua

Como Git e GitHub se conectam com pipelines automáticos para testar, revisar e publicar código sem intervenção manual.

O que é CI/CD? É a automação que entra em ação quando você faz um git push. O sistema roda testes, verifica qualidade de código e — se tudo passar — entrega o software em produção automaticamente. Git é o gatilho; CI/CD é o resultado.

Conceitos
CI vs CD — a diferença
Sigla Nome O que faz
CI Continuous Integration A cada push, roda testes e validações automaticamente. Detecta problemas antes de chegar em main.
CD Continuous Delivery Código aprovado é preparado e entregue automaticamente para um ambiente (staging, produção).
CD Continuous Deployment Versão mais avançada do Delivery — o deploy em produção acontece automaticamente sem aprovação manual.
GitHub Actions
CI/CD nativo do GitHub — o mais usado no mercado
GitHub Actions permite criar pipelines automáticos diretamente no seu repositório. Você define os passos em arquivos .yml dentro da pasta .github/workflows/.
%mkdir -p .github/workflows
Cria a pasta de workflows no projeto
%touch .github/workflows/ci.yml
Cria o arquivo de pipeline CI

Exemplo mínimo de workflow: roda testes em Node.js a cada push na main ou PR aberto.

name: CI
on:
push:
branches: [main]
pull_request:
jobs:
test:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v4
- uses: actions/setup-node@v4
- run: npm install
- run: npm test
Conexão com Git Flow
Como CI/CD se encaixa nas branches do treinamento
Branch / Evento O que o CI/CD faz
feature/* — push Roda testes unitários automaticamente. PR não pode ser aprovado se falhar.
develop — push Roda testes de integração. Pode fazer deploy automático no ambiente de staging.
release/* — push Roda suite completa de testes. Equipe valida manualmente antes do merge.
main — merge Deploy automático em produção (Continuous Deployment) ou aguarda aprovação (Continuous Delivery).
hotfix/* — push Roda testes rápidos de regressão — foco em velocidade para resolver o incidente.

O ciclo completo profissional: branch → commit → push → CI valida automaticamente → Pull Request revisado pelo time → merge → CD faz o deploy. O Git conecta cada etapa.

Ferramentas populares
Além do GitHub Actions
Ferramenta Característica
GitHub Actions Nativo do GitHub. Zero configuração de infra. Recomendado para quem já usa GitHub.
GitLab CI/CD Nativo do GitLab. Pipeline definido em .gitlab-ci.yml. Muito usado em empresas.
Jenkins Open source. Extremamente configurável. Comum em ambientes enterprise legados.
CircleCI Rápido e fácil de configurar. Plano gratuito generoso. Integra bem com GitHub.
Vercel / Netlify CD automático para projetos frontend. Cada push na main → deploy instantâneo.

GitHub CLI

Tudo que você faria no navegador do GitHub, feito direto pelo terminal com o comando gh.

O que é o GitHub CLI? É uma ferramenta oficial do GitHub que permite criar repositórios, abrir Pull Requests, gerenciar issues, fazer clone e muito mais — sem sair do terminal. Você já tem instalado e configurado.

Repositórios
Criar, clonar e renomear

Cria um repositório público no GitHub — use --private para privado
%gh repo create nome-do-repo --public --source=. --push
Cria o repo no GitHub E já envia o código local de uma vez
%gh repo clone usuario/nome-do-repo
Clona um repositório do GitHub para a máquina
%gh repo rename novo-nome
Renomeia o repositório do diretório atual no GitHub
%gh repo view --web
Abre o repositório atual no navegador
Pull Requests

Abre um Pull Request da branch atual para a branch principal
%gh pr create --web
Abre o formulário de PR no navegador com os dados já preenchidos

Lista os Pull Requests abertos no repositório
%gh pr checkout 42
Faz checkout local do PR número 42 para revisar o código
%gh pr merge 42 --squash
Mescla o PR comprimindo todos os commits em um só
Issues
Gerenciar tarefas e bugs pelo terminal
%gh issue create --title "Bug: descrição" --body "detalhes"
Cria uma nova issue no repositório
%gh issue list
Lista as issues abertas
%gh issue close 15
Fecha a issue número 15
Fluxo completo
Do zero ao PR sem abrir o navegador — como você fez no renomear
Este é o fluxo que você usou para renomear o repositório. Agora documentado no guia.
%cd nome-do-projeto
Entra na pasta
%gh repo rename novo-nome
Renomeia no GitHub
%git remote set-url origin https://github.com/seu-usuario/novo-nome.git
Atualiza o endereço remoto local para o novo nome
%git remote -v
Confirma que o remoto aponta para o novo endereço
%cd .. && mv nome-antigo novo-nome && cd novo-nome
Renomeia a pasta local para manter consistência (opcional)

Verificação final: rode git pull — se funcionar sem erro, tudo está conectado corretamente.

Referência rápida

Todos os comandos essenciais em um só lugar.

Fluxo do dia a dia
Comando O que faz
git add . Adiciona todos os arquivos ao stage
git commit -m "msg" Salva com mensagem
git push Envia para o GitHub
git pull Baixa e mescla mudanças
git status Estado atual do repo
Criar repositório
Comando O que faz
git clone URL Clona repo do GitHub (Fluxo A)
git init Cria repo local (Fluxo B e C)
git remote add origin URL Conecta ao GitHub
git push -u origin main Primeiro push (Fluxo B)
gh repo create nome --public --source=. --push Cria e publica tudo no terminal (Fluxo C)
Branches
Comando O que faz
git switch -c nome Cria e muda de branch
git merge nome Mescla branch
git branch -d nome Deleta branch
git stash / pop Guarda / restaura mudanças
Desfazer
Comando O que faz
git restore arquivo Descarta mudanças locais
git reset --soft HEAD~1 Desfaz commit, mantém stage
git revert HEAD Reverte commit publicado (seguro)
git reflog Histórico completo do HEAD
